\section{Architecture Système}

Le système repose sur une architecture web moderne organisée en couches, garantissant une séparation claire des responsabilités, une sécurité renforcée et une maintenance facilitée à long terme.

\subsection{Architecture en couches}

\begin{figure}[H]
    \centering
    \pumlsvg[0.8\textwidth]{arch_layers}
    \caption{Architecture générale en trois couches}
\end{figure}

L'application suit le pattern \textbf{MVC (Modèle-Vue-Contrôleur)} adapté au Web :
\begin{itemize}[label=\textcolor{primary}{\textbullet}]
    \item \textbf{Vue (View) :} Assurée par le Frontend React, elle gère l'interface utilisateur et l'expérience interactive de manière asynchrone.
    \item \textbf{Contrôleur (Controller) :} Les contrôleurs Spring Boot exposent des endpoints REST, reçoivent les requêtes du client et orchestrent les réponses.
    \item \textbf{Modèle (Model) :} Comprend la logique métier (Services), l'accès aux données (Repositories) et la structure de persistance MySQL.
\end{itemize}

\subsection{Sécurité et Authentification JWT}
\label{subsec:securite_jwt}

Pour répondre aux exigences de sécurité et de performance définies dans le cahier des charges, le système implémente une authentification \textbf{stateless} (sans état) basée sur les \textbf{JSON Web Tokens (JWT)} :

\begin{itemize}[label=\textcolor{primary}{\textbullet}]
    \item \textbf{Génération :} Lors de la connexion, le serveur valide les identifiants et génère un token signé cryptographiquement contenant l'identité et le rôle de l'utilisateur.
    \item \textbf{Transmission :} Ce token est renvoyé au client (React) qui le stocke localement et l'inclut dans le header \texttt{Authorization} de chaque requête API ultérieure.
    \item \textbf{Validation :} Le backend (Spring Security) intercepte chaque requête, vérifie la signature du token et extrait les droits d'accès sans avoir à interroger la base de données pour la session, garantissant ainsi une sécurité optimale et une grande rapidité de traitement.
\end{itemize}

\section{Conception Statique}
\label{sec:statique}

Le diagramme de classes constitue le pivot de la conception statique. Il définit la structure de données interne et les règles métier qui régissent les interactions entre les objets du système. Cette modélisation assure la cohérence des informations tout au long du cycle de vie d'une soutenance.

\begin{figure}[H]
    \centering
    \pumlsvg[\textwidth]{class_global}
    \caption{Diagramme de classes global du système}
\end{figure}

Les entités clés identifiées sont les suivantes :
\begin{itemize}[label=\textcolor{primary}{\textbullet}]
    \item \textbf{Utilisateur :} Classe de base gérant l'identité, les identifiants de connexion (email/password) et les droits d'accès (rôles).
    \item \textbf{Soutenance :} Entité centrale du système, elle fait le lien entre un projet, un jury, une salle et un créneau temporel précis au sein d'une session.
    \item \textbf{Projet :} Représente le travail académique (PFE, Mémoire, Stage) associé à un étudiant et supervisé par un enseignant encadrant.
    \item \textbf{Jury :} Structure d'association complexe définissant les rôles spécifiques (Président, Rapporteur, Examinateur) et gérant la présence effective des membres.
\end{itemize}

\subsection{Modèle de convocation officielle}
\label{subsec:convocation_doc}

La convocation est le document administratif pivot du système. Bien plus qu'une simple notification, elle constitue l'acte officiel qui déclenche la phase d'exécution des soutenances. Ce document est le lien contractuel entre l'institution, l'étudiant et les membres du jury.

L'automatisation complète de ce document via le \textbf{Document Service} apporte plusieurs garanties critiques :
\begin{itemize}
    \item \textbf{Intégrité absolue des données :} En extrayant les informations directement de la base de données après validation des conflits, le système élimine tout risque d'erreur humaine (inversion de salle, erreur de créneau ou faute de frappe dans les noms).
    \item \textbf{Valeur administrative :} Elle sert de justificatif officiel d'absence pour l'étudiant auprès de son employeur ou de son administration, et de convocation de mission pour les enseignants.
    \item \textbf{Réactivité et fluidité :} La génération est instantanée dès le verrouillage de la session par le coordinateur, permettant une diffusion massive et immédiate par voie électronique.
\end{itemize}
